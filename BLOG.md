# Web Audio/Video Analysis in the Browser: Videograms + Spectrograms

This project is a small, static web app that turns a local video file into two compact “timelines” you can see and scrub:

- **Videogram**: a spatiotemporal view of the video (100px tall, full width)
- **Spectrogram**: a time–frequency view of the audio (100px tall, full width)

Both visualizations are clickable, so you can **jump to any moment** in the video by clicking on either image. Each visualization can also be **downloaded as a PNG**.

## Why this is useful

When you’re exploring a recording, you often want a fast overview before you know what you’re looking for.

- The **videogram** makes large-scale changes in the image (cuts, motion, lighting changes) show up as patterns across time.
- The **spectrogram** highlights structure in the sound (speech, music, noise bursts, silence) in a way that’s easy to scan.

Together, they act like a lightweight “map” for scrubbing through video.

## The UI: player + two 100px timelines

The app UI is intentionally simple:

- A standard **HTML `<video>` player** for playback
- A **videogram canvas** below the player
- A **spectrogram canvas** below the videogram
- Download buttons for each canvas

Both canvases are **100px tall** and **fill the available width**, so they behave like compact timelines rather than large charts.

## Interaction: click-to-seek

The key interaction is that both the videogram and spectrogram behave like a seek bar:

1. You click at horizontal position \(x\) within the canvas.
2. That \(x\) position is mapped to a fraction of the total duration.
3. The video’s `currentTime` is updated to the corresponding timestamp.

A thin vertical playhead line is drawn over each visualization to show the current playback position.

## Videogram: “row-average over time”

The videogram is generated by sampling the video at many time points, and for each sampled frame:

1. **Downsample the frame** to a small working resolution (fast to process).
2. For every row of pixels, compute the **average luminance** across that row.
3. Treat those row averages as a 1D column and place it into the output image.

Stacking these columns from left to right over time produces a compact spatiotemporal signature of the entire video.

### Why the videogram is fast enough

The app uses a couple of practical tricks to keep videogram generation responsive:

- **Frame downsampling**: because we only need row averages, we don’t need full-resolution frames.
- **Adaptive sampling**: long videos don’t need one sample per screen pixel to be useful, so the app limits the number of time samples (and therefore the number of seeks).
- **`fastSeek()` when available**: some browsers can seek faster using `video.fastSeek(t)`.

## Spectrogram: sampled STFT in the browser

The spectrogram is generated from the video’s audio track using a Short-Time Fourier Transform (STFT):

1. **Decode audio** from the selected file in the browser.
2. **Downsample** the audio to a lower sample rate (mono) for faster FFT processing.
3. Compute FFT magnitudes across time windows.

### A key performance optimization: compute only what you draw

For a 100px-tall spectrogram, the horizontal resolution is typically the bottleneck (how many time slices you compute).

Instead of computing an STFT frame for every hop across the entire audio, the app:

- Chooses a target number of **time-columns** to render (capped),
- Computes FFT magnitudes for **only those columns**, and
- Renders/scales the result to fill the canvas.

This keeps spectrogram generation fast even for longer recordings.

## Export: download PNGs

Both canvases can be exported with a single click. The app uses the canvas `toDataURL("image/png")` method and triggers a download with a temporary `<a download>` link.

## Privacy and security

This app is entirely client-side:

- The file is selected locally using a file picker.
- Processing happens in the browser.
- Nothing is uploaded anywhere by the app itself.

## Run it locally

Because it’s a static site, you can serve it with any static server:

```bash
python3 -m http.server 5173
```

Then open `http://localhost:5173` and choose a video file.

## Deploy on GitHub Pages

The repository includes a GitHub Actions workflow that deploys the static site to GitHub Pages on push to `main`.

To enable it:

1. Open your repo in GitHub.
2. Go to **Settings → Pages**.
3. Set **Source** to **GitHub Actions**.

After the workflow runs, the app will be available at your repository’s Pages URL.

## Where to go next

If you want to extend this project, good next steps include:

- Adding a **settings panel** for sample rate / FFT size / color maps
- Adding **zooming** and selection ranges on the canvases
- Exporting raw data (CSV) in addition to PNG
- Using `requestVideoFrameCallback` (where available) to improve frame sampling behavior
